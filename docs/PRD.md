# PRD: 반도체 시료 생산주문관리 시스템 (SampleOrderSystem)

- 문서 출처: `[CRA_AI] Day3_개인과제_반도체시료관리.pdf` Chapter 01(개인과제 개요), Chapter 02(기능 명세) — 4~23페이지
- 상세 기능별 스펙은 [`FEATURES/`](FEATURES) 디렉터리 참고
- 개발 가이드는 [`CLAUDE.md`](../CLAUDE.md) 참고

## 1. 배경

가상의 반도체 회사 "S-Semi"는 다양한 반도체 시료(Sample)를 생산하여 연구소, 팹리스(Fabless) 업체,
대학 연구실 등에 납품한다. 시료는 주문이 들어오면 웨이퍼 공정 설비를 통해 제작되고, 검수 후 고객에게 출고된다.

최근 주문량이 급증하면서 다음과 같은 문제가 발생했다.

- "이 주문 처리됐는지" 확인이 어렵다.
- 공정 예약 후 완성 시점을 알 수 없다.
- 재고가 충분한데도 불필요한 추가 공정이 진행된다.

원인은 엑셀·메모장 기반의 수기 관리로, 실수가 잦고 재고·공정 현황을 한눈에 파악할 수 없다는 데 있다.
이를 해결하기 위해 **반도체 시료 생산주문관리 시스템**을 개발한다.

## 2. 목표

- 시료 등록/조회/검색을 체계적으로 관리한다.
- 고객 주문의 접수 → 승인/거절 → (필요 시 생산) → 출고까지의 전체 흐름을 상태 기반으로 추적한다.
- 재고 부족 시 자동으로 생산라인에 등록하고, 생산 완료 시 자동으로 주문 상태를 갱신한다.
- 담당자가 현재 주문/재고/생산 현황을 한눈에 파악할 수 있는 모니터링 기능을 제공한다.
- 콘솔 기반 UI로 동작하며, 담당자가 직접 명령(메뉴 번호)을 입력해 조작한다.

## 3. 역할(Actor) 및 흐름

| 역할 | 책임 |
|---|---|
| 고객 (시료 요청자) | 필요한 시료를 이메일 등으로 요청 (시스템 밖 프로세스) |
| 주문 담당자 | 요청에 맞게 주문서 작성(시료 예약), 승인된 주문의 출고 처리 |
| 생산 담당자 | 개발 시료 등록, 주문 수신 후 승인 또는 거절 처리 |

흐름: `고객 → (시료 요청) → 주문 담당자 → (주문서 전달) → 생산 담당자 → (승인/거절) → 주문 담당자`

이 시스템은 위 세 역할을 사람이 아니라 **하나의 콘솔 애플리케이션의 메뉴들**로 구현한다(역할별 로그인 분리는 요구되지 않음).

## 4. 시스템 개요

### 4.1 생산 라인

- 생산 라인은 공장에서 시료 하나를 생산하는 설비 흐름이다.
- 하나의 생산 라인은 시료를 한 번에 하나씩 생산한다 (단일 라인, 순차 처리).
- 주문이 들어온 시료에 대해서만 생산한다 (재고 예측 생산 없음).

### 4.2 실행 방식

- 콘솔 기반으로 동작하며 담당자가 직접 메뉴 번호 등을 입력한다.
- 시료를 등록하고 주문을 처리하는 것이 핵심 동작이다.

### 4.3 전체 프로세스 (Swimlane)

```
[주문 담당]  주문 등록(RESERVED)
                 │
[생산 담당]  승인 여부 판단
     ├─ 거절 ──────────────────────► REJECTED (정상 흐름 제외, 모니터링 대상 아님)
     └─ 승인
           │
        재고 확인
     ├─ 재고 충분 ─────► 출고 준비(CONFIRMED) ─► 출고 처리(RELEASE) ─► 고객 도달
     └─ 재고 부족 ─────► 생산 요청(PRODUCING) ─(생산 완료)─► 출고 준비(CONFIRMED) ─► 출고 처리(RELEASE) ─► 고객 도달
```

## 5. 주문 상태 정의

모든 주문은 아래 5개 상태 중 하나를 갖는다.

| 상태 | 의미 |
|---|---|
| `RESERVED` | 주문 접수 |
| `REJECTED` | 주문 거절 (정상 흐름 외의 상태이며 **모니터링에서 제외**) |
| `PRODUCING` | 주문 승인 완료 및 재고 부족으로 생산 중 |
| `CONFIRMED` | 주문 승인 완료 및 출고 대기 중 |
| `RELEASE` | 출고 완료 |

상태 전이 규칙:

- `RESERVED` → (승인, 재고 충분) → `CONFIRMED`
- `RESERVED` → (승인, 재고 부족) → `PRODUCING`
- `RESERVED` → (거절) → `REJECTED`
- `PRODUCING` → (생산 완료) → `CONFIRMED`
- `CONFIRMED` → (출고 처리) → `RELEASE`
- `REJECTED`/`RELEASE`는 종단 상태이며 그 외 전이는 없다.

## 7. 핵심 도메인 규칙 요약

- **시료(Sample)**: 시스템의 가장 기본 단위. 속성은 시료 ID, 이름, 평균 생산시간, 수율. 등록된 시료만 주문 가능.
- **수율(Yield)**: `정상적인 시료 수 / 총 생산 시료 수`. 예: 100개 생산 중 90개 정상 → 수율 0.9.
- **실 생산량**: `ceil(부족분 / 수율)` — 불량률을 고려해 부족분보다 더 많이 생산.
- **총 생산 시간**: `평균 생산시간 × 실 생산량`.
- **생산 큐 스케줄링**: FIFO(선입선출).
- **재고 상태 분류** (모니터링에서 사용):
  - 여유: 주문 대비 재고 충분
  - 부족: 주문 대비 재고 수량 부족
  - 고갈: 재고 수량이 0

## 8. 비기능 요구사항 / 개발 방침

- **아키텍처**: Model/View/Controller 패키지 구조 및 역할 분리 (`ConsoleMVC` PoC 계승)
- **데이터 영속성**: 애플리케이션을 재실행해도 데이터 유지 (`DataPersistence` PoC의 JSON 저장 구조 계승)
- **데이터 모니터링 Tool**: 저장된 데이터 상태를 콘솔에서 실시간 조회 (`DataMonitor` PoC 계승/통합)
- **Dummy 데이터 생성 Tool**: 테스트용 더미 주문 데이터를 생성해 연결된 저장소에 추가 (`DummyDataGenerator` PoC 계승/통합)
- **Agentic Engineering 주안점** (미션2 평가 기준):
  1. `CLAUDE.md`, `docs/PRD.md` 등 문서 관리
  2. Harness(작업 절차/스킬) 도입
  3. Test (핵심 도메인 로직 단위 테스트)
  4. Clean Code
  5. Commit 이력 관리

## 9. Out of Scope (본 과제 범위 밖)

- 다중 생산 라인 동시 운영 (본 시스템은 단일 생산 라인을 전제로 한다. PDF 예시 화면에는 "생산라인 1개(단일 라인)"로 명시됨)
- 사용자 인증/권한 분리 (역할별 로그인 없음, 콘솔 담당자가 모든 메뉴에 접근 가능)
- 네트워크/다중 사용자 동시 접속

## 10. 미해결/확인 필요 사항

- PDF 8페이지 표에는 출고 완료 상태 명칭이 `RELEASE`로 표기되어 있으나, 7페이지 다이어그램에는 `RELEASED`로 표기됨.
